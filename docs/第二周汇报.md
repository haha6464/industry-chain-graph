# Agent 化产业链图谱本周汇报

汇报日期：2026-06-26

## 1. 本周目标

本周工作围绕“产业链图谱展示 demo”向“可构建、可校验、可维护的产业链图谱 Agent”升级展开，重点目标包括：

1. 将图谱数据从展示型 `graph.json` 扩展为可审计、可导出、可复核的数据格式。
2. 接入阿里云百炼 Qwen Responses API，实现自动联网搜索、抽取和语义校验。
3. 建立构建、校验、更新、导出、应用候选图谱的完整工作流。
4. 将前端从单一图谱展示页拆分为“图谱展示”和“Agent 工作流”两类页面。
5. 明确后续 25 个基础行业批量生产的输入范围和交付路径。

## 2. 本周已完成工作

### 2.1 数据标准与行业池

- 扩展 `graph.json` 字段，支持 Agent 所需的审计信息：
  - `node_type`
  - `tags`
  - `business_description`
  - `is_key_node`
  - `chain_segment`
  - `source_urls`
  - `evidence_ids`
  - `confidence`
  - `updated_at`
- 明确当前版本不涉及公司信息：
  - 不抽取公司节点。
  - 不保留 `company_list`。
  - 不处理股票代码、财务指标和个股相关内容。
- 根据 mentor 表格补齐 25 个基础行业 manifest，当前前端可以选择完整行业池。
- 图谱层级从“上游/中游/下游”调整为数字 `level`：
  - `level=0` 为行业根节点。
  - `level=1` 为一级产业链环节。
  - `level=2/3/4/5...` 为逐级细分的产品、原料、工艺、设备、渠道、应用场景等节点。
- 上游/下游只保留在节点间关系或 `chain_position` 位置标签中，不再作为层级命名。

### 2.2 Agent 构建链路

- 删除 manual URL 模式，构建 Agent 改为自动联网搜索。
- 接入阿里云百炼 Qwen Responses API：
  - 使用 `web_search`
  - 使用 `web_extractor`
  - 使用 `code_interpreter`
  - 支持 `agent_max` 搜索策略。
- 构建流程当前包括：
  1. 生成搜索计划 `search_plan.json`
  2. 百炼联网搜索并抽取候选图谱
  3. 输出 `pre_validation_candidate_graph.json`
  4. 硬规则预校验
  5. 百炼语义校验与最小修图
  6. 输出 `candidate_graph.json`
  7. 输出证据库 `sources.jsonl`
  8. 输出校验报告、构建报告和复核队列
  9. 导出 mentor 格式 CSV
- Prompt 已根据本周测试结果优化：
  - 目标深度约 5-6 层。
  - 目标节点数量 60-100 个。
  - 节点数量不超过 150 个。
  - 避免只沿少数分支深挖，要求补齐横向分支。
  - `level=1` 通常不少于 5 个一级环节。
  - 重要一级环节尽量展开 3-8 个二级/三级分支。

### 2.3 校验与候选图谱应用

- 实现硬规则校验，覆盖：
  - 重复节点 ID
  - 重复节点名称
  - 缺失来源 URL
  - 低置信度节点/关系
  - 关系类型非法
  - 关系起点/终点缺失
  - 同一节点对存在多种主关系
  - 孤立节点
  - 公司字段误入
  - level 非法
- 新增规模与广度 warning：
  - 节点数低于 60：提示图谱广度不足。
  - 节点数超过 150：提示可能过细，需要人工复核。
  - `level=1` 少于 5 个：提示一级环节覆盖不足。
- 百炼校验 Agent 支持最小程度修改候选图谱：
  - 修正明显错误层级。
  - 修正明显错误方向。
  - 合并明显重复或同义节点。
  - 不大规模重写图谱。
  - 证据不足时进入复核队列。
- 新增“应用候选图谱”能力：
  - `candidate_graph.json` 可在校验通过后应用为正式 `graph.json`。
  - `update_candidate_graph.json` 可在校验通过后应用为正式 `graph.json`。
  - 应用后自动刷新 CSV。
  - 更新候选图谱应用时，会尝试合并新增 `checked_sources` 到 `sources.jsonl`。

### 2.4 增量更新 Agent

- 实现更新维护 Agent MVP。
- 支持三类运行模式：
  - `check_only`
  - `propose`
  - `apply`
- 当前增量更新逻辑遵循：
  - 没有足够新增证据则输出 `no_change`。
  - 默认不直接修改正式图谱。
  - 新增或修改必须保留 URL 来源。
  - 第一版不直接删除节点，只标记废弃或进入复核。
- 输出产物包括：
  - `update_proposal.json`
  - `update_candidate_graph.json`
  - `update_report.md`
  - `update_agent_request_prompt.txt`
  - `update_agent_raw_response.txt`

### 2.5 前后端能力升级

- 后端新增 Agent 相关 API：
  - 启动构建
  - 启动更新
  - 生成搜索计划
  - 运行校验
  - 导出 CSV
  - 查看 Agent 产物
  - 查看 run 状态
  - 中断 run
  - 应用候选图谱
- 构建和更新改为后端子进程运行：
  - 前端不再只显示 loading。
  - 可以查看当前步骤、命令和实时日志。
  - 支持中断耗时任务。
- 前端拆分为两个页面：
  - 图谱展示页：负责行业选择、图谱展示、筛选、节点审计、问答。
  - Agent 工作流页：负责搜索规划、构建、校验、更新、应用候选、导出和产物查看。
- Agent 运行监控改为右侧抽屉：
  - 显示 run id、状态、当前步骤。
  - 显示实际执行命令。
  - 显示实时日志。
  - 支持中断运行。
- 图谱展示侧适配新版格式：
  - 前端直接读取正式 `graph.json`。
  - 不再需要先导入 Neo4j 样例。
  - 初始页面不自动加载食品饮料图谱。
  - 选择行业后再加载正式图谱。
- 节点审计模块优化：
  - 显示 level，而不是上中下游层级。
  - 显示父节点和子节点。
  - 父子节点支持点击跳转。
  - 来源 URL 单行省略。
  - 邻接关系使用“隶属于”“是……的下游”的中文表达。
  - 审计面板有独立滚动条。

## 3. 当前可演示内容

本次汇报可以现场演示以下路径：

1. 打开前端，进入“Agent 工作流”页。
2. 选择某个行业。
3. 生成搜索计划，查看 `search_plan.json`。
4. 启动构建 Agent，右侧抽屉查看当前步骤、命令和日志。
5. 查看 Agent 产物：
   - request prompt
   - raw response
   - candidate graph
   - validation report

   - CSV

8. 点击节点查看节点审计、父子节点、来源 URL 和邻接关系。
9. 运行增量更新，查看 `update_proposal.json` 和 `update_candidate_graph.json`。
10. 应用更新候选图谱并刷新正式图谱。

## 4. 当前问题与风险

### 4.1 Agent 输出稳定性仍需继续调优

当前 Agent 已能自动联网搜索并生成候选图谱，但不同产业的资料结构差异较大，输出质量会受来源网页、搜索结果、模型抽取稳定性的影响。

已做控制：

- 强制 URL 来源。
- 限制关系类型。
- 排除公司信息。
- 硬规则校验。
- 百炼语义校验与最小修图。
- 候选图谱不自动覆盖正式图谱。

后续仍需通过多行业试跑继续调 prompt 和规则。

### 4.2 图谱广度需要重点关注

近期测试中，图谱深度基本够，但横向覆盖不足，容易形成少数分支深挖。

本周已调整：

- Prompt 增加 60-100 节点目标。
- Prompt 增加 150 节点上限。
- Prompt 要求一级环节和兄弟分支覆盖。
- 校验报告增加节点数量和一级环节覆盖 warning。

后续需要通过 2-3 个行业试点验证是否改善。

### 4.3 正式应用仍需要人工确认

目前系统支持将候选图谱应用为正式图谱，但仍建议保留人工确认：

- 候选图谱可能存在语义争议。
- 某些行业节点命名粒度需要研究员判断。
- 增量更新不应为了“更新”而制造节点。

因此当前设计仍是：

候选图谱 -> 校验报告/复核队列 -> 人工确认 -> 正式 graph.json

### 4.4 批量 25 行业还未完成

当前已经完成行业池登记和工作流能力建设，但 25 个行业标准数据包尚未批量跑完。

下一阶段重点是：

- 选择 2-3 个非食品饮料行业试点。
- 固定跨行业 prompt 和校验阈值。
- 再批量构建 25 个行业。
